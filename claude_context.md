# Context for Claude: Complexe Zaken TOB-Backoffice (Phase 3)

Hello Claude! You are stepping into an active project that we are currently transforming from a static React parser app into a fully-fledged, stateful GIS Backoffice. 

Here is the current state of architecture and the roadmap for what you need to build next.

## 1. Architectural Blueprint
- **Frontend Stack:** React (Vite), React Router, Context API (for Auth).
- **Backend Stack:** Supabase (PostgreSQL, Auth, RLS).
- **Goal:** Allow an `admin` to upload and parse documents (PDF/DOCX/Excel) containing location data, save this data to the database, and share specific projects with `external` users.

## 2. What is already finished (Phases 1 & 2)
1. **The Supabase Database:** Fully deployed. It has 5 tables with Row Level Security (RLS) enabled:
   - `profiles` (linked to `auth.users`, has a `role` column which is either 'admin' or 'external').
   - `projects` (main projects / TOB reports).
   - `locations` (extracted addresses/coordinates, linked to `project_id`).
   - `researches` (investigation statuses, linked to `location_id`).
   - `project_members` (links an `external` user to a specific `project_id` via RLS).
2. **Authentication:** The React app now has an `AuthContext` (`src/context/AuthContext.jsx`) that manages the Supabase session and fetches the user's role.
3. **Routing:** `App.jsx` has been refactored into a `BrowserRouter`. Unauthenticated users are forced to the `/login` route. Authenticated users are routed to the `Dashboard.jsx`.
4. **Environment Setup:** `@supabase/supabase-js` is installed, and the client is configured in `src/utils/supabaseClient.js`.

## 3. What YOU need to build next (Phase 3: Database Integration)
You are now in **Phase 3**. The old parsing logic in `Dashboard.jsx` still maps data to an *in-memory array* (`setLocations(finalLocations)`). We need to connect this to Supabase.

**Your Tasks:**
1. **Backend Service Layer (`src/services/api.js` or similar):** Create the CRUD functions for Supabase:
    - `saveProject(name, client)` -> returns new Project ID.
    - `saveLocations(projectId, locationsArray)` -> batch insert.
    - `saveResearches(locationId, defaultVinkjes)` -> batch insert.
    - `fetchProjects()`, `fetchLocations(projectId)`, `fetchResearches(locationId)`.
2. **Refactoring the Parser Flow:** When the user uploads a PDF/Word file in `Dashboard.jsx` and clicks "Process", intercept the final array generated by the parser. Instead of instantly showing it on a map, use your Service Layer to **INSERT the data into Supabase** as a new Project with its Locations.
3. **Updating the UI:** The `DataPreview.jsx` and Dashboard should now read their state from the Supabase database via `useEffect` queries, not from the local memory array.

**Important Note on File Structure:** 
Check `src/pages/Dashboard.jsx` for the current React state. The parser tools (`src/utils/pdfParser.js` etc.) work perfectly fine, you only need to change what happens *after* the parsing is complete.

Good luck!
